# PR Auto-Checklist (Danger.js)

**Status:** Aktiv siden 2026-05-08
**Eier:** Engineering / PM
**Kilde:** `dangerfile.ts` i repo-rot
**CI-jobb:** `.github/workflows/danger.yml`

## Hva dette er

En Danger.js-bot som kjører på hver PR mot `main` og kommenterer en
context-sensitive review-sjekkliste basert på hvilke filer som endres.
Eks:

- Hvis du rører `apps/backend/src/wallet/` → bot minner om outbox-pattern
  og hash-chain-integrity.
- Hvis du rører en migration → bot minner om idempotente
  `CREATE TABLE IF NOT EXISTS` per ADR-012.
- Hvis du rører compliance-kode → bot minner om §11/§66/§71 og
  `gameType` MAIN_GAME vs DATABINGO.

Botens jobb er å plassere de viktigste sjekkpunktene foran reviewer
sine øyne mens PR-en ennå er fersk — ikke å erstatte review.

## Hvilke regler er aktive

Se `dangerfile.ts` for den autoritative kilden. Regelliste per
2026-05-08:

| # | Trigger | Type | Skill / Doc |
|---|---|---|---|
| 1 | Fundament-doc endret uten skill-endring | `warn()` | `.claude/skills/<name>` |
| 2 | `apps/backend/src/wallet/` eller `adapters/` endret | `message()` | `wallet-outbox-pattern` |
| 3 | Spill 1 master-kode (Game1MasterControlService m.fl.) | `warn()` | `spill1-master-flow` |
| 4 | Migration-fil i `apps/backend/migrations/` | `message()` | `database-migration-policy` (ADR-012) |
| 5 | `apps/backend/src/compliance/` endret | `warn()` | `pengespillforskriften-compliance` |
| 6 | Live-rom-arkitektur (Game1ScheduleTickService m.fl.) | `warn()` | `live-room-robusthet-mandate` (R1-R12) |
| 7 | PR-tittel ikke Conventional Commits | **`fail()`** | `ENGINEERING_WORKFLOW.md` §1 |
| 8 | PR-beskrivelse < 50 tegn | `warn()` | `.github/pull_request_template.md` |
| 9 | Kritisk kode (wallet/compliance/game) uten test-endringer | `warn()` | — |

Regel 7 er den eneste som blokker merge. Alle andre er info-level og
kan ignoreres hvis reviewer er enig.

## Hvordan bypasse (skip-danger)

Hvis PR-en din er en triviell rename eller doc-flytt der reglene ikke
gjelder, sett `[skip-danger]` i PR-beskrivelsen. Dette slår av regel
1-6 + 8-9.

**Regel 7 (Conventional Commits-format) kan IKKE bypasses** — det er en
hard infrastructure-regel fra `ENGINEERING_WORKFLOW.md`.

Eksempel:

```markdown
## Summary
Refactor: rename internal helper.

## Test plan
- npm test

[skip-danger]
```

## Hvordan legge til en ny regel

1. Åpne `dangerfile.ts`.
2. Identifiser trigger-condition (regex på filsti, PR-tittel, body, etc.).
3. Bruk én av tre output-typer:
   - **`message()`** — info-bokser (informasjon, ikke advarsel)
   - **`warn()`** — gul advarsel (handle hvis relevant)
   - **`fail()`** — rød blokker merge (bruk SPARSOMT — kun for
     ufravikelige infrastructure-regler)
4. Test lokalt med `npx danger pr <PR-URL>` (krever
   `DANGER_GITHUB_API_TOKEN` i miljøet).
5. Hvis regelen kan bypasses, wrappe i `if (!skipChecklist) { ... }`.
6. Oppdater tabellen i denne doc-en.

### Designprinsipper for nye regler

- **Vær spesifikk.** "Sjekk at koden er bra" → nei. "Verifiser at
  outbox-rad enqueue-es i samme TX som ledger-mutering" → ja.
- **Lenk til skill / doc.** Reviewer skal kunne klikke seg videre uten
  å gjette hvor regelen kommer fra.
- **`fail()` skal være sjeldent.** Bruk det kun for
  infrastructure-regler som ALDRI bør brytes (Conventional Commits er
  et OK eksempel — regulatorisk-cap-håndhevelse er det ikke, det skal
  fanges av tests).
- **Ikke spam.** Hvis 80% av PR-er triggrer regelen, vil reviewer
  begynne å ignorere bot-kommentarene. Vurder strammere trigger.

## Vedlikehold

- Test-PR for å verifisere bot-kommentar etter regelendringer.
- Hvis en skill endrer navn / flyttes → oppdater `dangerfile.ts`.
- Hvis et fundament-doc-navn endres (f.eks. ny audit-rapport) →
  oppdater `fundamentDocs`-arrayet.
- Hvis vi tar i bruk et nytt fundament-tema (f.eks. bonus-game-engine)
  → vurder ny regel.

## CI-integrasjon

- Workflow: `.github/workflows/danger.yml`
- Triggers: `pull_request` mot `main` (opened, synchronize, reopened,
  edited).
- Permissions: `contents: read` + `pull-requests: write`.
- Token: default `GITHUB_TOKEN` (ingen ekstra secret).
- Concurrency: én danger-jobb per PR; nye push-er kansellerer kjørende.

## Lokal testing

```bash
# Hvis du har en åpen PR du vil teste mot:
DANGER_GITHUB_API_TOKEN=<din-PAT> npx danger pr <PR-URL>

# Tørrkjøring uten å poste kommentar:
npx danger pr <PR-URL> --json
```

## Kjente begrensninger

- Botens kommentarer er på engelsk fra danger-rammeverket; vår
  innhold er på norsk. Format-headeren ("Generated by 🚫 Danger") er
  fast.
- Botens kommentarer overskrives ved hver kjøring (samme bot-comment-id),
  så historiske advarsler finnes ikke som separate kommentarer.
- `fail()` på PR-tittel kan kun fjernes ved å rebase commit-historikken
  for å rette tittelen — danger ser kun PR-tittel, ikke commit-meldinger.
