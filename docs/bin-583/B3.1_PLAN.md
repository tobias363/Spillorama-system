## 0. Open questions for reviewer (stopp her hvis svar trengs før kode)

1. **Rolle-modell** — Legg til ny `AGENT`-rolle på `app_users.role` CHECK, eller gjenbruk `HALL_OPERATOR`?
   - Forslag: ny `AGENT`. Legacy-agent har meningsfullt annen kapabilitet (shift-state, kontantkasse, wallet-nære ops). HALL_OPERATOR forblir «admin-personell per hall» (sees i eksisterende permissions).
   - Alternativ ved avslag: gjenbruk `HALL_OPERATOR`, lag `AGENT_*`-permissions som gater shift/kasse-endepunkter.

2. **Multi-hall per agent** — Legacy `agent.hall: []` gir agenten tilgang til flere haller. Ny `app_users.hall_id` er 1:1.
   - Forslag: ny m:n-tabell `app_agent_halls (user_id, hall_id, is_primary, assigned_at)`. `app_users.hall_id` settes til NULL for AGENT-rollen og reserveres for HALL_OPERATOR.
   - Konsekvens: shift-FK peker på `app_agent_halls` (eller bare `app_halls.id` + scope-sjekk).

3. **Hall-valg ved login** — Legacy matcher hall på request-IP mot `hall.ip` + `hall.agents[]`. Ny auth er token-basert og API-first.
   - Forslag: agent oppgir `hallId` i login-body (ny `POST /api/agent/auth/login`), eller eksplisitt `POST /api/agent/shift/start` etter standard `/api/auth/login`. Flagger IP-matching som «out-of-scope for B3.1, håndteres i terminal-registrering (annen PR)».

4. **Shift = implicit på login, eller explicit start?** — Legacy oppretter agentShift implisitt i `postLogin`. Ny design foreslår *explicit* shift-start via egen POST.
   - Forslag: explicit. Gir clean separation av auth (hvem) fra shift (når jobber du). Støtter også shift-pause/bytte-hall uten ny innlogging senere.

5. **Scope av shift-skjema i B3.1** — Skal vi porte HELE agentShift-skjema (inkl. cash-felter) nå, eller bare lifecycle-felter?
   - Forslag: port hele skjemaet (ref-struktur matcher legacy 1:1). Lifecycle-endepunkter eksponeres i B3.1; cash-mutering tas av B3.2/B3.3. Dette minimerer schema-churn senere.

6. **Default-språk** — Legacy `agent.language` default `'english'`. Norsk kontekst?
   - Forslag: default `'no'`, migrer eksisterende rader ved behov. Flagg for avklaring.

### 3.1 `20260418200000_agent_role_and_profile.sql`

```sql
-- BIN-583 B3.1: add AGENT role + agent-profile fields on app_users.
--
-- Agents are operators who run the physical hall's bingo kassa: they log
-- in, open a shift, sell physical tickets, handle cash in/out. Legacy
-- stored them in a separate 'agent' collection; we unify under app_users
-- with a new role so auth/profile infra is shared.

ALTER TABLE app_users
  DROP CONSTRAINT IF EXISTS app_users_role_check;

ALTER TABLE app_users
  ADD CONSTRAINT app_users_role_check
  CHECK (role IN ('ADMIN', 'HALL_OPERATOR', 'SUPPORT', 'PLAYER', 'AGENT'));

ALTER TABLE app_users
  ADD COLUMN IF NOT EXISTS language        TEXT NOT NULL DEFAULT 'no',
  ADD COLUMN IF NOT EXISTS avatar_filename TEXT NULL,
  ADD COLUMN IF NOT EXISTS parent_user_id  TEXT NULL
    REFERENCES app_users(id) ON DELETE SET NULL,
  ADD COLUMN IF NOT EXISTS agent_status    TEXT NOT NULL DEFAULT 'active'
    CHECK (agent_status IN ('active', 'inactive'));

COMMENT ON COLUMN app_users.language IS 'BIN-583: ISO-639-1 UI-språk. Default no for nytt oppsett.';
COMMENT ON COLUMN app_users.parent_user_id IS 'BIN-583: agent-hierarki (legacy parentId). NULL for non-AGENT.';
COMMENT ON COLUMN app_users.agent_status IS 'BIN-583: aktiv/inaktiv AGENT. inactive = kan ikke logge inn, men data bevart.';

CREATE INDEX IF NOT EXISTS idx_app_users_parent_user_id
  ON app_users(parent_user_id) WHERE parent_user_id IS NOT NULL;
CREATE INDEX IF NOT EXISTS idx_app_users_role_status
  ON app_users(role, agent_status) WHERE role = 'AGENT';
```

### 3.2 `20260418200100_agent_halls.sql`

```sql
-- BIN-583 B3.1: many-to-many between AGENT users and halls.
--
-- Legacy stored halls as an embedded array on the agent document. In
-- relational form this is a join table. An agent MUST have ≥1 hall to
-- start a shift; 'is_primary' marks which hall the UI defaults to.

CREATE TABLE IF NOT EXISTS app_agent_halls (
  user_id     TEXT NOT NULL REFERENCES app_users(id) ON DELETE CASCADE,
  hall_id     TEXT NOT NULL REFERENCES app_halls(id) ON DELETE CASCADE,
  is_primary  BOOLEAN NOT NULL DEFAULT false,
  assigned_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  assigned_by_user_id TEXT NULL REFERENCES app_users(id) ON DELETE SET NULL,
  PRIMARY KEY (user_id, hall_id)
);

CREATE INDEX IF NOT EXISTS idx_app_agent_halls_hall_id
  ON app_agent_halls(hall_id);

-- En agent kan kun ha én primary hall om gangen.
CREATE UNIQUE INDEX IF NOT EXISTS uniq_app_agent_halls_primary_per_user
  ON app_agent_halls(user_id) WHERE is_primary;
```

### 3.3 `20260418200200_agent_shifts.sql`

```sql
-- BIN-583 B3.1: agent shifts — one row per agent's work-session in a hall.
--
-- Ports legacy agentShift collection 1:1 including cash-settlement
-- columns so we don't schema-churn in B3.2/B3.3. Only lifecycle
-- endpoints (start/end/get-current/list) ship in B3.1; cash columns
-- remain at default until B3.2/B3.3 services start writing.

CREATE TABLE IF NOT EXISTS app_agent_shifts (
  id                            TEXT PRIMARY KEY,
  user_id                       TEXT NOT NULL REFERENCES app_users(id) ON DELETE RESTRICT,
  hall_id                       TEXT NOT NULL REFERENCES app_halls(id) ON DELETE RESTRICT,
  started_at                    TIMESTAMPTZ NOT NULL DEFAULT now(),
  ended_at                      TIMESTAMPTZ NULL,
  is_active                     BOOLEAN NOT NULL DEFAULT true,
  is_logged_out                 BOOLEAN NOT NULL DEFAULT false,
  is_daily_balance_transferred  BOOLEAN NOT NULL DEFAULT false,

  -- Cash-settlement felter (B3.2/B3.3 driver dem; B3.1 kun defaults):
  daily_balance                 NUMERIC(14, 2) NOT NULL DEFAULT 0,
  total_daily_balance_in        NUMERIC(14, 2) NOT NULL DEFAULT 0,
  total_cash_in                 NUMERIC(14, 2) NOT NULL DEFAULT 0,
  total_cash_out                NUMERIC(14, 2) NOT NULL DEFAULT 0,
  total_card_in                 NUMERIC(14, 2) NOT NULL DEFAULT 0,
  total_card_out                NUMERIC(14, 2) NOT NULL DEFAULT 0,
  selling_by_customer_number    INTEGER NOT NULL DEFAULT 0,
  hall_cash_balance             NUMERIC(14, 2) NOT NULL DEFAULT 0,
  hall_dropsafe_balance         NUMERIC(14, 2) NOT NULL DEFAULT 0,
  daily_difference              NUMERIC(14, 2) NOT NULL DEFAULT 0,
  control_daily_balance         JSONB NOT NULL DEFAULT '{}'::jsonb,
  settlement                    JSONB NOT NULL DEFAULT '{}'::jsonb,
  previous_settlement           JSONB NOT NULL DEFAULT '{}'::jsonb,

  created_at                    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at                    TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- En agent kan maks ha én aktiv shift om gangen (partial unique-index).
CREATE UNIQUE INDEX IF NOT EXISTS uniq_app_agent_shifts_active_per_user
  ON app_agent_shifts(user_id) WHERE is_active;

-- Rapport-lookup: hvilke agenter er aktive i en hall akkurat nå.
CREATE INDEX IF NOT EXISTS idx_app_agent_shifts_hall_active
  ON app_agent_shifts(hall_id, is_active) WHERE is_active;

-- Historisk oppslag per agent.
CREATE INDEX IF NOT EXISTS idx_app_agent_shifts_user_started
  ON app_agent_shifts(user_id, started_at DESC);

-- Historisk oppslag per hall (settlement-rapport).
CREATE INDEX IF NOT EXISTS idx_app_agent_shifts_hall_started
  ON app_agent_shifts(hall_id, started_at DESC);
```

## 7. Wallet-idempotency

B3.1 **skriver ikke** til wallet (alle kontantbevegelser er i B3.2/B3.3). Men vi etablerer nøkkel-konvensjonen som senere PR-er arver:

```
agent-shift:{shiftId}:start       → wallet-seed ved shift-start (B3.2)
agent-shift:{shiftId}:end         → wallet-settlement ved shift-end (B3.3)
agent-shift:{shiftId}:cash-in:{txId}  → kontant-innskudd (B3.2)
agent-shift:{shiftId}:cash-out:{txId} → kontant-uttak (B3.2)
```

I B3.1 lagres `shiftId` som TEXT (KSUID/ULID) — hvilket format gjenbrukes av B3.2 som idempotency-nøkkel-prefiks. B3.1-kode som bruker WalletAdapter (`wallet-seed` ved start) flyttes til B3.2.
