### Q4 вЂ” Cash-in payment-type: Cash vs Card (shift-balance-effekt)

Legacy skiller Cash/Card pГҐ shift:
- Cash в†’ pГҐvirker `agentShift.dailyBalance` (mГҐ vГ¦re в‰Ґ amount for cash-withdraw)
- Card в†’ gГҐr direkte pГҐ wallet, shift-balance uendret

- **Alternativ:** SlГҐ sammen til kun wallet-op (ignorer paymentMethod). Enklere kode men bryter paritet og gjГёr daily cash-oppgjГёr (B3.3) umulig.

### Q5 вЂ” Cancel-sale-vindu for physical ticket

Legacy har implisitt В«kan kansellere innen samme shiftВ» (via shift-end rollback). Skal B3.2 ha eksplisitt `POST /api/agent/physical/sell/cancel`?

- **Mitt forslag:** Ja, med 10-minutters vindu etter sell. Innen vinduet: refund wallet + ticket.status = 'AVAILABLE'. Utover: admin-force required (AGENT_SHIFT_FORCE permission).
- **Alternativ:** Drop cancel i B3.2; legg til i B3.3 settlement. Ulempe: agenter mГҐ bekrefte sale to ganger for ГҐ unngГҐ feil.

### Q6 вЂ” Transaction list: dag-skille (wall-clock vs shift)

`GET /api/agent/transactions/today` вЂ” hva er В«todayВ»?

- **Mitt forslag:** Shift-basert: alle transaksjoner med `shift_id = getCurrentShift()?.id` (eller siste shift hvis ingen aktiv). Matcher legacy.
- **Alternativ:** Wall-clock-basert (00:00-23:59 lokal tid). Enklere ГҐ forklare, men shift som starter kl 19:00 og avsluttes kl 02:00 splittes over 2 dager.

## 3. Nye tabeller вЂ” schema + migrasjoner

### 3.1 `20260418230000_agent_transactions.sql`

```sql
-- BIN-583 B3.2: agent-transaction log вЂ” port of legacy agentTransaction.
--
-- One row per agent-initiated money-op (cash-in/out, ticket sale, ticket
-- registration). Writes are append-only; corrections come via counter-
-- transactions (same pattern as wallet_transactions).
--
-- Columns mirror legacy agentTransaction, modernised:
--   - amounts in NUMERIC(14,2) (same as wallet_accounts)
--   - payment_method enum instead of free-text "paymentBy"
--   - typeOfTransaction в†’ action_type enum (DEPOSIT, WITHDRAW,
--     TICKET_SALE, TICKET_REGISTER, TICKET_CANCEL, FEE, OTHER)
--   - category (debit/credit) derived вЂ” stored as wallet_direction enum
--   - wallet_tx_id FK til wallet_transactions(id) for traceability
--   - shift_id FK til app_agent_shifts(id) for per-shift rapportering

CREATE TABLE IF NOT EXISTS app_agent_transactions (
  id                  TEXT PRIMARY KEY,
  shift_id            TEXT NOT NULL REFERENCES app_agent_shifts(id) ON DELETE RESTRICT,
  agent_user_id       TEXT NOT NULL REFERENCES app_users(id) ON DELETE RESTRICT,
  player_user_id      TEXT NOT NULL REFERENCES app_users(id) ON DELETE RESTRICT,
  hall_id             TEXT NOT NULL REFERENCES app_halls(id) ON DELETE RESTRICT,
  action_type         TEXT NOT NULL CHECK (action_type IN (
                        'CASH_IN', 'CASH_OUT',
                        'TICKET_SALE', 'TICKET_REGISTER', 'TICKET_CANCEL',
                        'FEE', 'OTHER'
                      )),
  wallet_direction    TEXT NOT NULL CHECK (wallet_direction IN ('CREDIT', 'DEBIT')),
  payment_method      TEXT NOT NULL CHECK (payment_method IN ('CASH', 'CARD', 'WALLET')),
  amount              NUMERIC(14, 2) NOT NULL CHECK (amount >= 0),
  previous_balance    NUMERIC(14, 2) NOT NULL,
  after_balance       NUMERIC(14, 2) NOT NULL,
  wallet_tx_id        TEXT NULL,          -- Wallet-transaction-ID hvis wallet-op utfГёrt
  ticket_reference    TEXT NULL,          -- unique_id / player_ticket_id for ticket-rader
  external_reference  TEXT NULL,          -- For card-PSP-id el.l. (valgfritt)
  notes               TEXT NULL,
  other_data          JSONB NOT NULL DEFAULT '{}'::jsonb,
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),

  -- Cancelled transactions get counter-row with related_tx_id pointing
  -- to the original. Original row is NOT mutated.
  related_tx_id       TEXT NULL REFERENCES app_agent_transactions(id) ON DELETE SET NULL
);

-- Rapport-indekser:
CREATE INDEX IF NOT EXISTS idx_app_agent_tx_shift
  ON app_agent_transactions(shift_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_app_agent_tx_agent
  ON app_agent_transactions(agent_user_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_app_agent_tx_player
  ON app_agent_transactions(player_user_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_app_agent_tx_hall_created
  ON app_agent_transactions(hall_id, created_at DESC);
-- Daglig aggregert rapport (typisk brukt av B3.3 settlement):
CREATE INDEX IF NOT EXISTS idx_app_agent_tx_shift_action
  ON app_agent_transactions(shift_id, action_type, payment_method);

COMMENT ON TABLE app_agent_transactions IS
  'BIN-583: append-only log av agent-initierte transaksjoner. Port av legacy agentTransaction.';
COMMENT ON COLUMN app_agent_transactions.wallet_tx_id IS
  'Wallet-transaction-ID (wallet_transactions.id) hvis wallet-op utfГёrt. NULL for rene kontant-rader uten wallet-effekt.';
COMMENT ON COLUMN app_agent_transactions.related_tx_id IS
  'Hvis dette er en kansellering, peker pГҐ opprinnelig tx. Original-raden muteres ikke (append-only).';
```

## 8. Shift cash-column-mutation

Hver cash-in/out-operasjon muterer `app_agent_shifts`:

| Action | Column change |
|---|---|
| Cash-in (paymentMethod=CASH) | `total_cash_in += amount`, `daily_balance += amount` |
| Cash-out (paymentMethod=CASH) | `total_cash_out += amount`, `daily_balance -= amount` |
| Cash-in (paymentMethod=CARD) | `total_card_in += amount` |
| Cash-out (paymentMethod=CARD) | `total_card_out += amount` |
| Ticket-sale (paymentMethod=CASH) | `total_cash_in += price`, `daily_balance += price`, `selling_by_customer_number += 1` |
| Ticket-sale (paymentMethod=WALLET) | Shift unchanged вЂ” kun wallet-debit |

Invariant: `daily_balance в‰Ґ 0` ved cash-out. Fail-closed med `INSUFFICIENT_DAILY_BALANCE`.

Alle mutations gjГёres i samme DB-transaksjon som `app_agent_transactions`-INSERT + wallet-call (via `client.query('BEGIN'...)`).
